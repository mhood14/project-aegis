Networking

Hub-and-spoke networking with deliberate traffic paths.

The network design keeps shared services, application hosting, AI services, and data services from collapsing into one flat trust zone. Private endpoints, DNS decisions, and egress routing all matter here.

Reference architecture diagram

Open full diagram

Core layout

  • Hub VNet: shared routing functions, private DNS, and central network coordination live here.
  • App spoke: App Service uses VNet integration so the Flask app can reach downstream services through controlled paths.
  • AI spoke: Azure OpenAI and Azure AI Search are reached through private endpoint patterns.
  • Data spoke: Blob Storage stays aligned to the document workflow rather than broad public reachability.

Egress strategy

  • Accurate current design: outbound inspection is planned through an NVA built from a Linux machine, not Azure Firewall.
  • Traffic direction: user-defined routes can steer spoke traffic toward the NVA path for centralized control.
  • Why it matters: this keeps the egress story aligned with the project’s real implementation choice and cost-conscious design.

Flow

How traffic and trust move through the networked system.

01

User reaches the app

The user authenticates through Entra ID before the application begins serving document or retrieval functionality.

02

App reaches private services

App Service uses its integrated network path to reach storage, search, and model services without relying on broad public service exposure.

03

Private DNS keeps names honest

Private endpoint routing only works cleanly when name resolution follows the intended private path instead of leaking to public resolution.

04

NVA controls outbound direction

The Linux-based NVA provides the egress control point for traffic that should leave through an inspected, centralized route.
