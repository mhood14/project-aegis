Logging & Testing

How the project records behavior and pressure-tests trust boundaries.

Aegis treats logging as part of the architecture, not just a debugging convenience. The app emits meaningful events, Azure diagnostics can flow into Log Analytics, and Sentinel can surface suspicious patterns for review.

What gets tracked

Activity across the system can be followed instead of guessed at.

📝

Application events

The Flask app logs events like authorization allowed, authorization denied, search indexing, LLM requests started, content sanitization, content filter blocks, and request failures.

📊

Platform telemetry

Azure-side diagnostics from services such as App Service, Key Vault, and other resources can be sent into Log Analytics, where Sentinel analytics rules can work from the collected data.

Sequence walkthroughs

Four concrete examples of what happens and what can be queried later.

01

Prompt injection attempt

A malicious document or retrieved chunk includes lines such as “ignore previous instructions” or tries to suppress citations. Before the model call, the app sanitizes matching lines from retrieved content. If risky material is removed, the app logs a retrieved_content_sanitized event. If everything useful is stripped out, the request can be blocked before it reaches the model. If Azure OpenAI itself blocks the prompt, the app records an llm_request_blocked_content_filter event.

02

Repeated failed authorizations

A user repeatedly requests a scope they are not allowed to access. The app logs authorization_denied events with the requested scope, user context, and denial reason. Those denials can be forwarded into Log Analytics and picked up by a Sentinel analytics rule that looks for repeated failures over time or unusual access patterns from the same identity.

03

Querying the evidence afterward

After a test or incident, the project supports querying the log trail to answer basic questions: Which request ID triggered the block? Which document IDs were involved? Did the app sanitize retrieved content? Was the model request started, completed, or blocked? Did the same user generate repeated denials? This makes the system easier to investigate after the fact instead of relying on memory.

AppServiceConsoleLogs
| where ResultDescription contains 'authorization_denied'
| summarize count() by CallerIpAddress, bin(TimeGenerated, 15m)

04

Tracing activity across the system

Logs can show how one workflow moves across layers: a user authenticates, the app evaluates scope, documents are retrieved, sanitization may occur, a model call starts, the answer completes or is blocked, and the resulting telemetry can be reviewed in Log Analytics or Sentinel. That end-to-end trail is important because AI incidents rarely live in just one service.

Example detection queries

AppServiceConsoleLogs
| where ResultDescription contains 'retrieved_content_sanitized'
AppServiceConsoleLogs
| where ResultDescription contains 'llm_request_blocked_content_filter'
AppServiceConsoleLogs
| where ResultDescription contains 'authorization_denied'
| summarize attempts=count() by ResultDescription, bin(TimeGenerated, 1h)
