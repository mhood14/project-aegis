Interview Flow

How to walk the project through with a security engineering team.

This page turns the project into a discussion tool: start with the architecture, move into identity and control logic, then use the threat model and telemetry story to show engineering depth rather than feature breadth.

01

Start with the architecture, not the UI

Lead with the hub-and-spoke model, private service placement, trust zones, and why the app, AI, data, and secrets were separated.

02

Explain identity and authorization first

Walk through Entra ID SSO, Conditional Access, RBAC, and why scope enforcement is the critical trust control for the application.

03

Show the document and retrieval lifecycle

Discuss upload, metadata tagging, chunking, storage, hybrid retrieval, and how only scoped context is sent into the model flow.

04

Use the threat model as the differentiator

Highlight prompt injection, poisoning, authorization abuse, and why AI trust issues were treated as application security problems.

05

Close with detections and maturity

Connect cloud and application logs to Sentinel, then explain how the design supports monitoring, incident response, and future hardening.

Why this matters

It frames the project around security engineering judgment.

Instead of presenting the work like “I built an AI app,” this flow helps present it as “I designed a secure cloud system with AI-specific trust controls and operational visibility.” That reads much better for Microsoft-aligned roles.
