Architecture

Hub-and-spoke Azure architecture for a private AI workflow.

Project Aegis separates shared network services, application hosting, AI services, and data services so that identity, routing, private connectivity, and observability can be managed deliberately rather than as a single flat boundary.

Network layout

  • Hub VNet: shared routing functions, private DNS, and central control-plane services.
  • App spoke: the Flask application runs in App Service with VNet integration for controlled service access.
  • AI spoke: Azure OpenAI and Azure AI Search are reached through private endpoint paths.
  • Data spoke: Blob Storage access is isolated from broad public exposure and aligned with the retrieval workflow.

Egress direction

  • Phase 1: native Azure outbound routing keeps the initial build cost lower while preserving the segmented layout.
  • Target state: Azure Firewall in the hub, UDRs on spoke subnets, forwarded traffic enabled in peering, and centralized inspection through the firewall path.
  • Reason for the phased approach: establish the security architecture first, then add forced tunneling and centralized outbound enforcement as the environment matures.

Core flow

How identity, data, and model access move through the system.

01

User signs in with Entra ID

Authentication begins with Entra ID, while Conditional Access and MFA add policy enforcement before the application session is trusted.

02

Application applies authorization

The Flask backend validates scope and access rules before allowing uploads, retrieval, or downstream model use.

03

Documents are stored and indexed

Uploaded source documents are stored in Blob Storage, processed into chunks, and prepared for Azure AI Search retrieval.

04

Scoped context reaches Azure OpenAI

Only the retrieval output selected for the current request is sent forward into the model path, with citations tied back to chunk sources.

Provisioning baseline

The build checklist shows the environment starting from management groups, role assignments, resource groups, VNets, subnets, peerings, private DNS, NSGs, Entra ID P1 licensing, Conditional Access, Key Vault, Defender for Key Vault, and diagnostic settings.

Why the separation matters

By keeping application logic, search, model access, secrets, and storage as distinct responsibilities, the environment avoids treating the model itself as the main trust boundary. Identity, routing, and application logic still do the hard security work.
