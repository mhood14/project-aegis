Executive Summary

A resume artifact built to prove security judgment, not just cloud familiarity.

This project began as a full Microsoft tenant and secure AI application build, then evolved into a case study centered on the controls, trust boundaries, and risks that matter in enterprise environments. The result is a portfolio artifact that demonstrates how identity, networking, application logic, and AI safety controls have to work together.

What was built

A Flask-based document upload and question-answering application hosted in Azure, protected by Entra ID SSO, integrated with Azure OpenAI and Azure AI Search, and backed by private storage and secrets management.

What was secured

Identity flows, scope-aware authorization, east-west network paths, private service exposure, application secrets, AI retrieval boundaries, and security telemetry pipelines into Sentinel and Log Analytics.

Why it matters

The project shows practical understanding of Zero Trust principles in an AI setting, where network isolation alone is not enough and the real trust problem lives in authorization, retrieval, and data handling.

Network pattern

  • Hub VNet: central routing, NVA traffic forwarding, private DNS, and shared control-plane functions.
  • App spoke: Flask App Service with VNet integration and isolated workload placement.
  • AI spoke: Azure OpenAI and Azure AI Search via private endpoint paths.
  • Data spoke: storage account access isolated from the public internet.

Control intent

  • Keep model access private and application-mediated.
  • Separate app, AI, and data roles so trust decisions do not collapse into a single service boundary.
  • Use private DNS and bidirectional peering to preserve internal service reachability without public exposure.
  • Force security relevance into the architecture, not just the README.

Security Decisions

The interesting part is not the stack. It is why each control exists.

Identity and Access

  • Entra ID used for SSO and tenant-backed user identity.
  • Conditional Access policies require MFA and block legacy auth.
  • Users, groups, and RBAC roles are designed as durable access controls, not one-off assignments.
  • High-risk security flaw identified in the threat model: scope handling must be authoritative and derived from trusted claims.

Secrets and Service Authentication

  • Azure Key Vault stores application secrets and environment-sensitive values.
  • Managed identity reduces dependence on hardcoded credentials.
  • Defender for Key Vault and diagnostic settings strengthen visibility on secret access patterns.
  • Public access is removed where possible to reduce unnecessary exposure.

Data and AI Safety

  • Documents are treated as untrusted input, not trusted instructions.
  • Chunking reduces blast radius and supports narrower retrieval context.
  • Prompt injection and document poisoning are explicitly modeled as primary risks.
  • Retrieved content should remain separated from control instructions in orchestration logic.

Monitoring and Detection

  • Subscription and activity logs flow into Log Analytics and Microsoft Sentinel.
  • Application logs are treated as security telemetry, not just troubleshooting noise.
  • Detection concepts include authz denial spikes, suspicious prompt activity, fallback anomalies, and abnormal scope usage.
  • The design supports incident response and future validation of AI misuse cases.

Threat Model

The strongest signal in this project: security thinking was documented, prioritized, and mapped to controls.

The executive threat model identifies a critical takeaway: in AI systems, network isolation is necessary but not sufficient. The real control problem sits in trust management across uploads, retrieval, prompt assembly, authorization, and output handling.

Critical

Authorization / scope abuse

Unauthorized access to higher-sensitivity content if scope logic is weak or user-controlled.

Priority action: derive scope authoritatively from Entra claims, groups, or policy.
High

Prompt injection via retrieved content

Malicious instructions embedded in documents can manipulate answers or attempt hidden instruction following.

Priority action: isolate retrieved context, harden orchestration prompts, and flag suspicious chunks.
High

Knowledge poisoning

Adversarial or low-trust content inside a valid scope can degrade integrity and commercial usefulness.

Priority action: quarantine, provenance tracking, review states, ignore controls, and rollback capability.
Med-High

Infrastructure misconfiguration

Private DNS, NVA, routing, and east-west connectivity are powerful controls, but also operational dependencies.

Priority action: validate route tables, DNS behavior, and segmentation continuously.

Interview Walkthrough

How I would present this project to a security engineering team.

01

Start with the architecture, not the UI

Explain the hub-and-spoke model, private service placement, and why trust boundaries were separated before discussing any AI features.

02

Show identity and authorization first

Walk through Entra ID SSO, Conditional Access, and why backend scope enforcement is the key control for protecting sensitive content.

03

Demonstrate the document lifecycle

Upload, metadata tagging, chunking, indexing, storage placement, and retrieval with scoped context only.

04

Use the threat model as the differentiator

Discuss prompt injection, poisoning, over-retrieval, and how the project treats AI risk as an application trust problem rather than a marketing feature.

05

Close with detections and operational maturity

Show how logs feed Sentinel, where KQL detections fit, and how incident response and quarantine workflows could evolve the platform further.
