Azure • Entra ID • Private Endpoints • Azure OpenAI • AI Search • Sentinel

Enterprise AI with explicit identity, network, and retrieval controls.

Project Aegis is a private Azure AI environment built around a Flask application, Entra ID single sign-on, hub-and-spoke networking, Azure OpenAI, Azure AI Search, Blob Storage, Key Vault, and Microsoft Sentinel. The project focuses on keeping document access, model access, and operational visibility under enforceable control.

Review architecture Review controls
  • Hub-and-spoke Azure design with private DNS and private endpoint connectivity across app, AI, and data services.
  • Entra ID SSO, Conditional Access, RBAC, and server-side scope enforcement for document retrieval and application access.
  • Chunked document processing, scoped search results, citations, and Sentinel-ready telemetry for security monitoring.
Core stack Azure App Service, Azure OpenAI, Azure AI Search, Blob Storage, Key Vault, Log Analytics, and Sentinel.
Identity Entra ID SSO with MFA, group-based access, and tenant-backed authorization boundaries.
Application Flask retrieval application using managed identity, document uploads, chunk storage, and cited responses.
Operations Centralized diagnostics and security telemetry designed for Sentinel investigations and future detections.

System overview

What the project includes.

The environment starts with Azure management structure, resource groups, virtual networks, subnets, peering, private DNS, and NSGs. On top of that foundation, the application layer uses Entra ID for sign-in, stores secrets in Key Vault, keeps documents in Blob Storage, indexes chunked content for retrieval, and sends only scoped context into Azure OpenAI.

Platform foundation

Management groups, resource groups, hub-and-spoke VNets, subnet segmentation, NSGs, peering, and private DNS establish the deployment boundary.

Control priorities

Security decisions the implementation is built around.

Authoritative identity

User identity comes from Entra ID, with Conditional Access and MFA reducing the risk of weak or unmanaged authentication paths.

Private service access

Core cloud services are intended to stay behind private networking, private endpoints, and internal name resolution rather than broad public exposure.

Scoped retrieval

Documents are chunked and indexed for retrieval, but authorization remains an application responsibility rather than something delegated to the model.

Operational visibility

Diagnostics feed Log Analytics and Sentinel so misuse, failures, and suspicious content workflows can be investigated instead of guessed at later.

Key outcomes

The short technical summary.

  • Built a secure Azure AI environment that separates application, AI, data, and shared network responsibilities across hub and spoke networks.
  • Implemented Entra ID sign-in, Conditional Access, MFA, groups, RBAC, and server-side scope controls to govern access to documents and retrieval results.
  • Used managed identity, Key Vault, Blob Storage, Azure AI Search, and Azure OpenAI to support upload, chunking, retrieval, and cited answer generation.
  • Documented major threats including authorization abuse, prompt injection, knowledge poisoning, application-layer weaknesses, and infrastructure misconfiguration.
  • Integrated diagnostics with Log Analytics and Sentinel so the environment supports monitoring, alerting, and future detection engineering.
