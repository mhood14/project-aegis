Security

Controls, trust boundaries, and AI-specific failure modes.

The core idea behind Project Aegis is that enterprise AI security is not solved by network isolation alone. It depends on how identity, retrieval, authorization, content trust, and monitoring are designed together as one control system.

Identity and access

  • Entra ID used for SSO and tenant-backed user identity.
  • Conditional Access requires MFA and blocks legacy authentication.
  • Users, groups, and RBAC roles are designed as durable control structures.
  • Scope enforcement must be derived from trusted claims and validated server-side.

Secrets and service auth

  • Azure Key Vault stores application secrets and sensitive configuration values.
  • Managed identity reduces hardcoded credential exposure.
  • Diagnostic visibility supports secret-access monitoring and review.
  • Private exposure patterns reduce unnecessary internet reachability.

Data and AI safety

  • Retrieved documents are treated as untrusted content.
  • Chunking narrows blast radius and retrieval scope.
  • Prompt injection and document poisoning are treated as primary design threats.
  • Context should stay separated from instruction logic in orchestration.

Monitoring and detection

  • Cloud and application events flow into Log Analytics and Sentinel.
  • Security-relevant application logs are treated as telemetry, not just debugging output.
  • Design supports alerts for suspicious prompts, authorization failures, and abnormal usage patterns.
  • Monitoring closes the loop between prevention and operational response.

Threat model

Highest-priority risks identified in the project.

The executive threat model prioritizes authorization abuse, prompt injection, poisoning, and infrastructure weakness as the key areas that can break trust in the system.

Critical

Authorization / scope abuse

Unauthorized access to higher-sensitivity content if scope logic is weak or user-controlled.

Priority action: derive scope from claims, groups, or policy rather than trusting user input.
High

Prompt injection

Malicious instructions embedded in retrieved documents can distort model behavior and outputs.

Priority action: isolate retrieved context and harden orchestration prompts.
High

Knowledge poisoning

Adversarial or low-trust data inside a valid scope can degrade output integrity and business value.

Priority action: add provenance, review state, quarantine, and rollback mechanisms.
Med-High

Infrastructure misconfiguration

Private DNS, NVA routing, and segmentation introduce operational dependencies that must be validated.

Priority action: regularly test routes, DNS, east-west paths, and segmentation boundaries.

Adversarial examples included

Supporting documents include test prompts meant to simulate malicious or manipulative retrieved content.

  • Ignore previous instructions
  • Fake authorization claims
  • Secret exfiltration requests
  • Evidence suppression attempts
