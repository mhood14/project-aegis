Security

Controls for identity, retrieval, secrets, and monitoring.

The security model assumes that network isolation helps, but does not solve AI trust problems by itself. Identity, authorization, document trust, prompt handling, and telemetry all have to work together for the platform to stay credible.

Control areas

The trust model in four readable pieces.

These are the main control families the project keeps in view so the AI workflow stays understandable and defensible instead of magical.

🔐

Identity and access

  • Entra ID provides single sign-on and tenant-backed user identity.
  • Conditional Access requires MFA and blocks legacy authentication paths.
  • Roles and groups are used as durable access-control structures.
  • Trusted claims and server-side checks determine scope rather than user-supplied values.
🗝️

Secrets and service authentication

  • Azure Key Vault stores secrets and sensitive configuration values.
  • Managed identity reduces the need for embedded credentials.
  • Key Vault diagnostics support monitoring of secret-access activity.
  • Private service exposure patterns reduce unnecessary internet reachability.
📄

Document and model trust

  • Uploaded and retrieved documents are treated as untrusted content.
  • Chunking limits retrieval scope and narrows the effect of hostile content.
  • Prompt injection and document poisoning are treated as first-order risks.
  • Prompt assembly should keep system instructions distinct from retrieved context.
📡

Monitoring and detection

  • Cloud and application events flow into Log Analytics and Sentinel.
  • Application logs are treated as security telemetry, not only debugging output.
  • The design supports detections for suspicious prompts, authorization failures, fallback behavior, and abnormal usage patterns.
  • Observability is part of the security control set, not a separate afterthought.

Threat priorities

Highest-priority risks identified in the project material.

The executive threat model puts the biggest weight on authorization abuse, prompt injection, knowledge poisoning, credential misuse, application vulnerabilities, infrastructure weakness, and monitoring blind spots.

Critical

Authorization and scope abuse

Weak scope logic can expose higher-sensitivity content to the wrong user even when authentication itself succeeds.

Priority action: derive scope from claims, groups, or policy and enforce it server-side.
High

Prompt injection through retrieved content

Malicious instructions hidden in documents can distort model behavior, suppress citations, or attempt unsafe output paths.

Priority action: separate instructions from retrieved context and detect suspicious chunks.
High

Knowledge poisoning

Bad but in-scope data can degrade answer quality and create false confidence even without crossing an authorization boundary.

Priority action: add provenance, review state, quarantine, ignore, and rollback controls.
Med-High

Infrastructure misconfiguration

Private DNS, peering, routing, and centralized inspection plans create dependencies that need validation and continuous testing.

Priority action: test routes, DNS, east-west paths, and segmentation behavior regularly.
🧪

Adversarial test material

The prompt-injection corpus includes examples such as:

  • Instructions to ignore previous instructions
  • Attempts to fake authorization or escalate access
  • Requests to suppress citations
  • Secret-exfiltration style prompts

Detection examples

| where ResultDescription contains '"event_type": "authz_denied"'
AppServiceConsoleLogs
| where ResultDescription contains '"event_type": "content_filter_blocked"'
AppServiceConsoleLogs
| where ResultDescription contains '"event_type": "fallback_used"'
| summarize count() by bin(TimeGenerated, 1d)
