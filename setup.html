Azure Setup

The cloud baseline behind the project.

This part of Aegis focuses on how the Azure environment is organized before the AI workflow even starts: identity, governance, secrets, diagnostics, and the deployment footing that the rest of the project depends on.

🏢

Tenant and governance baseline

The provisioning record shows management structure, resource organization, role assignments, and the supporting Azure layout needed to keep the project understandable as it grows.

🛂

Identity first

Entra ID provides the sign-in layer, with MFA and Conditional Access helping ensure that authentication policy is enforced before the app treats a session as trusted.

🗝️

Secrets and service trust

Azure Key Vault stores secrets while managed identity reduces the amount of long-lived credentials embedded in application code or configuration.

📈

Diagnostics as part of the build

Diagnostic settings are not treated as optional polish. They are part of the initial platform setup so later investigation has something real to work with.

What this setup enables

The Azure foundation supports the rest of the project in a few specific ways.

  • •
    Identity-backed sessions: the application can make authorization decisions using trusted claims and roles instead of user-supplied scope values.
  • •
    Safer service access: managed identity and Key Vault reduce casual secret sprawl.
  • •
    Operational evidence: diagnostics and monitoring data can be routed into Log Analytics and surfaced in Sentinel.
  • •
    Room for segmentation: governance and resource structure make the network and application pieces easier to separate cleanly.
